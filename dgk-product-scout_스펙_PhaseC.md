## C-목적

와이퍼·에어컨필터의 **차종별 검색 수요**를 데이터로 측정해, 신차 판매량 프록시가 놓치는 두 수요를 잡는다:
1. **구형 고수요차** — 보유대수 크고 보증 끝나 자가교체·가격민감(온라인 애프터마켓 핵심). "아반떼MD가 쏘렌토MQ4보다 부품 수요 클 수도" 케이스.
2. **떠오르는 신차** — 우리가 호환 제품을 보유했는데 미처 전용 등록 안 해 놓친 수요 케이스.

**호환 확인·상품 등록·소싱 판단은 사람(시경/팀)**. 시스템은 **수요 발굴만** 한다.

## C-원칙 (Phase A/B에서 피 흘려 배운 것 — 반복 금지)

1. **규모와 추세는 별도 컬럼.** 단일 매력도 점수로 합치지 말 것(임의 가중치 금지).
2. **mock ≠ 실제.** C-2 수확 실험으로 라이브 검증 후 본 파이프라인 구축. 수확이 차종을 충분히 주는지부터 눈으로 확인.
3. **사전은 인식용 — Phase B 함정과 본질이 다름.** 청소기 소모품 사전이 함정이었던 건 "무엇이 *관련 있는가*"(무한·가변)를 사전으로 정하려 해서임. 차종 사전은 "무엇이 *한국 자동차 모델인가*"(유한·닫힌 집합). 신차 나오면 한 줄 추가가 전부. **시드용 아님 — 수확된 토큰 중 진짜 차종을 걸러내는 인식·정규화 전용.**
4. **추측 숫자 금지.** 포함 최소규모·추세 윈도 등은 config에 `[확인 필요]`. C-2 결과 보고 확정.
5. **검산 1회 필수.**

## C-데이터 소스 역할 (Phase B와 동일 분담)

- **검색광고 키워드도구** = 절대 규모. 모델별 부품 검색량. (호출 규약은 `search_volume.py`에 이미 검증됨 — 변경 금지.)
- **데이터랩** = 추세(상대값). 모델별 부품 키워드의 시간 추세. (상대값 전용, 절대 규모 불가.)

## C-아키텍처: 꼬리물기 발견 + 사전 인식

```
[부품 시드] → 키워드도구 → relKeyword 수확
                              ↓
        부품어 제거 → 토큰을 차종 사전과 대조 (인식·정규화)
                              ↓
        인식된 모델별로 그 모델+부품 검색량 합산 (search_volume.py 합산 재사용)
                              ↓
    출력1: 규모 랭킹 (모델|부품유형|합산검색량)   [MVP]
    출력2: 규모 상위·신규 후보 → 데이터랩 추세 → 추세 랭킹   [후속]
```

- **부품 시드(config)**: 에어컨필터 계열 = `자동차에어컨필터`, `에어컨필터`, `캐빈필터` / 와이퍼 계열 = `와이퍼`, `와이퍼블레이드`. (≤5/호출. C-2에서 어느 시드가 모델을 잘 주는지 측정해 조정.)
- **정규화**: `아반떼MD`/`더뉴아반떼`/`올뉴아반떼AD` 등 별칭을 사전의 **정규명**으로 변환. **세대는 분리 유지**(아반떼MD와 아반떼AD는 다른 부품 = 다른 상품). 세대 없는 bare `아반떼`는 별도 모호 버킷(사람이 해석).
- **비매칭 토큰**(교체주기·정품·가격·삼성서비스센터…)은 버림.

## ★C-정직한 한계 (출력 상단에 명시할 것)

- **추세는 *이미 검색량이 쌓인* 모델만 잡는다.** 출시 직후 검색량 바닥인 진짜 따끈한 신차는 바닥을 찍고 올라오기 전까진 데이터에 안 보임(지연·하한). **그 구간은 시경의 시장감각이 데이터보다 빠르다 — 시스템은 보완재이지 대체재가 아니다.**
- **데이터랩 추세는 작은 모델일수록 노이즈.** 중~대형 모델에서만 신뢰.
- **검색 수요 ≠ 우리 판매량.** 시장 수요 우선순위 신호일 뿐.

## C-구현 단계

### C-1. 사전 로드 + 정규화
- 시경 교정본 `차종_인식사전_초안.md`(정규명 ↔ 별칭)를 데이터 파일(예: `data/car_models.*`)로 변환·로드.
- 정규화 함수: 입력 키워드 → 부품어 제거 → 별칭 매칭 → 정규명 반환(없으면 None).
- 초안으로 골격부터 만들고, 교정본 오면 데이터 파일만 교체.

### C-2. ★검증 게이트 (여기서 정지·보고)
부품 시드로 수확 실험. **본 파이프라인 만들기 전, 수확이 쓸 만한지 라이브로 확인.** 출력:
1. 인식된 distinct 모델 수.
2. 모델별 합산 규모 **상위 30** (정규명 | 부품유형 | 합산검색량 | 멤버수).
3. **수확 중 모델로 인식된 비율 vs 버려진 비율.**
4. **버려진 토큰 중 "모델처럼 보이는" 상위 30** — 사전 누락 차종을 드러내기 위함(시경이 이걸 보고 사전 보강).
→ **정지.** 이 결과로 ① 수확이 모델을 충분히 주는가 ② 사전 구멍은 어디인가를 판단. 수확이 빈약하면 *사전 모델 직접 질의*("{모델} 에어컨필터" 직접 호출)로 보강하는 길이 있으나 **API 호출 폭증 — 기본 아님, C-2 결과 보고 결정.**

### C-3. 규모 랭킹 본 파이프라인 (MVP)
- C-2 통과 시 정식 파이프라인. 출력1(규모 랭킹) 완성. **여기까지면 핵심 가치(구형 고수요차 발굴) 전달.**
- config: `MODEL_MIN_VOLUME = [확인 필요 — C-2 후]` (랭킹에 올릴 최소 규모. 꼬리 노이즈 컷.)

### C-4. 추세 랭킹 (후속)
- 규모 상위 + 신규 후보 모델의 `"{정규명} {부품}"`을 데이터랩에 태움(≤5그룹/호출 배치).
- 추세 = 최근 N개월 평균 / 과거 평균(또는 기울기). 상승률 컬럼.
- config: `TREND_RECENT_MONTHS`, `TREND_BASELINE_MONTHS` = `[확인 필요]`.

### C-5. 테스트 + 출력
- **unit test(fixture 스냅샷)**: ①별칭 정규화 정확(아반떼MD≠아반떼AD, bare 아반떼 분리) ②비모델 토큰 버림 ③모델별 합산 정확 ④규모 랭킹 정렬 ⑤추세 계산 정확 ⑥규모·추세 별도 컬럼(미합산) 확인.
- **반복 출력**: `모델 | 부품유형 | 절대규모 | 추세(상승률/방향)`. 단일 점수 없음. 정렬 규모순/추세순. 한계 주석 상단 표기.
