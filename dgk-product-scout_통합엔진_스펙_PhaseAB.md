# dgk-product-scout 통합엔진 스펙 — Phase A + B (Claude Code 인계용)

> **목적**: 계절 캘린더와 (후속) 차종 수요 스캐너를 *하나의 공통 코어* 위에 올리기 위한 구현 스펙.
> **이 문서 범위**: Phase A(공통 검색량+합산 코어) + Phase B(계절 캘린더 완성). **Phase C(차종 수요 스캐너)는 설계 결정 대기 중 — 본 문서 미포함**(맨 아래 참고).
> **빌드 순서**: A → B → C. 반드시 순서대로.
> **현재 HEAD**: `557d00f` (계절 캘린더 통합 랭킹). 재사용할 합산 로직은 `eacc698`.

---

## 0. 절대 원칙 (인수인계 6장 — 위반 금지)

1. **추측 숫자를 코드에 박지 말 것.** 미확정 임계값은 config에 `[확인 필요]` 주석으로 두고, 실측 보고 후 시경이 확정. (임의 하향/상향 금지)
2. **mock 통과 ≠ 실제 작동.** 라이브 API 응답을 눈으로 검증한 뒤 굳힐 것.
3. **단일 점수로 줄세우지 말 것** — 축은 별도 컬럼(임의 가중치 금지).
4. **부분 수정이 다른 부분을 깨뜨림.** 합산 로직을 코어로 옮길 때 기존 테스트 전수 통과 확인.
5. **검산 1회 필수** — 숫자 산출물은 교차검증.
6. **데이터랩=상대값**(절대 검색량 불가). 규모는 검색광고 키워드도구로.

## Phase A — 공통 검색량 + 합산 코어

### A-목표
현재 `naver_adapter.py`의 per-seed 기기군 합산 로직(`eacc698`에서 구현·테스트됨)을 **계절·차종 두 모듈이 함께 쓰는 재사용 함수로 추출**한다. 새로 발명하지 않는다 — 이동·정리만.

### A-1. 합산 함수 추출
- 신규 모듈 제안: `src/core/search_volume.py`
- 핵심 함수(시그니처는 기존 코드에 맞춰 조정):
  ```
  fetch_aggregated_volume(seed_keywords: list[str]) -> dict
  # 반환 구조(시드별):
  # {
  #   seed: {
  #     "total_volume": int,              # 기기군/카테고리 합산 (monthlyPcQcCnt + monthlyMobileQcCnt 의 멤버 전체 합)
  #     "member_keywords": list[{relKeyword, volume, compIdx, plAvgDepth}],
  #     "unattributed_candidates": list[...]   # 시드에 귀속 안 된 연관어 = 자체 후보
  #   }
  # }
  ```
- **시드당 1회 호출** 원칙 유지(`eacc698`의 핵심). 멀티시드 배치(5개/호출) 폐기 상태 유지 — substring 귀속 불가 때문.
- 시드 응답의 소모품 연관어 전부를 그 시드 멤버로 합산(cross-named 포함). 개별값은 `member_keywords`로 보존.

### A-2. 검색광고 키워드도구 호출 규약 (변경 금지 — 이미 검증됨)
- `GET https://api.searchad.naver.com/keywordstool`
- params: `hintKeywords`(≤5, 콤마구분), `showDetail=1`
- HMAC 서명: `msg = "{timestamp}.{method}.{path}"` (쿼리스트링 제외) → HMAC-SHA256(secret) → base64. `timestamp = round(time.time()*1000)`
- 헤더 4종: `X-Timestamp`, `X-API-KEY`, `X-Customer`, `X-Signature`
- 응답: `keywordList[{relKeyword, monthlyPcQcCnt, monthlyMobileQcCnt, plAvgDepth, compIdx}]`
- `compIdx` = 낮음/중간/높음. 저소량 `"< 10"` → `LOW_VOLUME_FALLBACK`(=0) 그대로.
- **규모 정의**: `monthlyPcQcCnt + monthlyMobileQcCnt`.

### A-3. 호출량 관리
- 키워드도구는 ≤5 hintKeywords/호출. 시드 많아지면 배치 + **캐시** 필요(동일 시드 재호출 방지). 캐시 TTL은 config.

### A-4. 테스트
- 기존 `eacc698` 라인 테스트(68개) 전수 통과 유지. 코어 추출 후 import 경로만 바뀌고 동작 동일해야 함.

## Phase B — 계절 캘린더 완성

### B-1. 규모 측정을 합산으로 전환 ★(시경 동의 완료)
- `seasonal_calendar.py`의 규모를 **단일 키워드 → Phase A 공통 코어 합산**으로 교체.
- **계절성은 그대로 데이터랩**(대표 키워드 1개, 계절성 지수 = 최대월/연평균). 역할 분담 유지: 데이터랩=계절성, 검색광고 합산=규모.
- **근거(왜 바꾸나)**: 현재 표의 와이퍼=19,430(단일)이 벤치마크 합산 179,730과 약 9배 차이. 차종 성공 시장 기준선(와이퍼 18만 / 에어컨필터 35만)과 자릿수가 달라 오판 발생. 합산으로 통일하면 같은 척도.

### B-2. 임계 규모 재보정 ★(추측 금지 — 실측 후 확정)
- 현재 `필터 규모 ≥ 20,000`은 **단일 키워드 척도 기준이라 폐기**. 합산으로 바꾸면 모든 숫자가 커지므로 임계도 재설정 필요.
- **절차**: ① Claude Code가 전 시드를 합산으로 재측정 → ② 결과 숫자를 **보고**(차종 벤치마크 와이퍼 18만/에어컨필터 35만 대비 어디쯤인지 포함) → ③ 시경이 임계 확정 → ④ config 입력.
- config: `MARKET_SIZE_THRESHOLD = [확인 필요 — 합산 재측정 후 확정]`
- **계절성 임계 2.0은 유지**(비율이라 척도 무관). `STABILITY_TOLERANCE_MONTHS = 1` 유지.

### B-3. 전 시즌 씨앗 목록 (아래는 ★초안 — 시경 교정 필요)
- 현재 테스트 키워드 11개뿐 → 전 시즌으로 확장해야 "전수 줄세움"이 됨.
- **주의**: 아래는 후보 키워드일 뿐, 계절성/규모 수치는 *데이터가 판정*(추측 입력 금지). 시스템이 각 키워드를 진짜계절(≥2.0)/약한(1.8~2.0)/상시(<1.8)로 분류.
- **상시 대조군**은 "상시로 정직 분류되는지" 검증용(와이퍼·워셔액이 그랬듯 분류 로직 정상 확인).

```
[여름] 햇빛가리개, 차량용선풍기, 통풍시트커버, 차량용쿨매트, 차량용방석, 김서림방지제
[장마] 발수코팅제, 유리발수, 김서림제거제, 차량용제습제
[겨울] 부동액, 냉각수, 성에제거제, 성에제거기, 성에커버, 스노우체인, 타이어체인,
       체인스프레이, 겨울용워셔액, 핸들커버, 열선시트커버
[봄·환절기] 꽃가루, 황사, (에어컨필터 — 상시지만 봄 가점 관찰용)
[상시 대조군] 와이퍼, 워셔액, 에어컨필터, 엔진오일
```
> 교정 포인트: 빠진 시즌상품 추가 / 시경 경험상 "이건 시장 거의 없음" 제거 / 명칭 변형(예: "썬블럭" vs "햇빛가리개") 추가 여부.

### B-4. 겨울 시즌코어 합산 자동 반영 (보류 a 해소)
- 기존엔 겨울 합산(~6.9만)을 단일 키워드 표에 "끼워넣기"가 척도 충돌이라 보류였음.
- **B-1으로 표 전체가 합산 척도가 되면 겨울도 같은 척도로 자연 포함** → 별도 끼워넣기 불필요. 보류 (a) 자동 해소.

### B-5. 계절 스크립트 unit test 추가 ★(시경 무오류 기준)
- 현재 계절 스크립트는 라이브 검증만, 별도 unit test 없음(인수인계 4장).
- **방식**: 실제 API 응답(데이터랩 + 키워드도구)을 **fixture로 스냅샷** → 계절성지수 계산·합산·랭킹·필터 로직을 **결정론적으로** 검증. (CSV 브리지 때 쓴 방식과 동일 — 라이브를 직접 단정 못 하니 응답을 고정.)
- 최소 케이스: ①계절성 지수 정확 계산 ②합산 정확(멤버 합) ③필터 경계(규모·계절성) ④정렬 옵션(규모순/계절성순) ⑤상시 대조군이 상시로 분류.

### B-6. 반복 실행 가능한 출력
- 통합 랭킹(시즌 무관 줄세움) 유지. 컬럼: `제품 | 시즌 | 계절성지수 | 정점/상승월 | 절대규모(합산) | 안정성`.
- **단일 매력도 점수 없음**(임의 가중치 금지). 정렬: 규모순(기본)/계절성순. 필터: `규모 ≥ [B-2 확정값] AND 계절성 ≥ 2.0`(기본 ON).
- 매 시즌 재실행 시 깔끔히 표만 갱신되도록(결과 표 출력 정리).

### B-검수 (Phase B 완료 판정)
- [ ] 와이퍼·에어컨필터를 합산으로 측정 시 벤치마크(18만/35만)와 같은 자릿수로 나오는가 (척도 정합 확인 = 검산)
- [ ] 상시 대조군이 상시로 분류되는가
- [ ] 임계값이 `[확인 필요]`가 아니라 시경 확정값으로 채워졌는가
- [ ] 계절 unit test 통과

## Phase C — 차종 수요 스캐너 (★결정 대기 — 본 문서 미포함)

확정된 설계:
- 출력 2종: **수요 규모 랭킹**(절대 검색량, 구형차 보유대수 케이스) + **상승 추세 랭킹**(데이터랩 증가율, 신차 케이스).
- 발견: 부품 시드("에어컨필터"·"와이퍼")의 연관어 꼬리물기로 차종 후보 수확.
- 같은 공통 코어(Phase A) 재사용.

**미결정(시경 결정 필요) — 차종 *인식* 방식**:
- 꼬리물기는 `"{X} 부품"`을 수확하지만 X가 차종인지 수식어("교체주기"·"정품"·"가격")인지 시스템이 못 가림.
- (i) **작은 차종 인식 사전**: 시스템이 차종만 골라 깔끔한 랭킹 출력. 사전 관리 소요 있음(작게 시작·점증). *시드용 아님 — 분류용.*
- (ii) **사전 없이 전체 키워드 랭킹**: 사람이 눈으로 차종 골라냄. 관리 0, 노이즈 있음.
- → 이 결정 후 Phase C 스펙 작성.
