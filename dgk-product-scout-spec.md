# dgk-product-scout — 신제품 발굴 자동화 시스템 설계서 (v2 · 2-레이어)

| 항목 | 내용 |
|---|---|
| 문서 목적 | Claude Code 구현 핸드오프용 설계 스펙 (최종본) |
| 구조 변경 | v1(단일 SKU 평가) → **v2(2-레이어: 카테고리 발굴 → SKU 검증)** |
| 작성 기준 | chat Claude(설계) → Claude Code(구현/테스트) 2-Claude 워크플로 |
| 제안 리포지토리 | `dltlrud93-spec/dgk-product-scout` |
| 기술 스택 | Python + Streamlit, SQLite 스냅샷, Claude API, Google Drive 백업 |
| 운영자 | 이시경(대표), 박민우(MD) 외 |

> **이 문서의 절대 원칙**
> 1. 추측한 숫자를 코드에 박지 않는다. 미확정 값은 설정 파일 파라미터로 빼고 `[확인 필요]` 표시.
> 2. 마진 계산은 새로 만들지 않는다. 기존 견적서 원가 계산 로직을 `cost_engine` 모듈로 재사용.
> 3. 인증·법적 판단은 **자문 플래그**일 뿐, 시스템이 확정 결론을 내리지 않는다.
> 4. 산출물은 "결정"이 아니라 **사람이 빠르게 베팅할 수 있는 근거**다.
> 5. **시스템은 2개 층이다. "어떤 제품군에 진입할까"(Layer 1)가 먼저, "그 안에서 무엇을·어떻게 팔까"(Layer 2)가 나중.**

---

## 1. 시스템 개요

### 1.1 2-레이어 정의
- **Layer 1 (카테고리 발굴)** — *가장 중요.* "어떤 제품군/시장에 진입할까"를 찾는다. 출력 = **카테고리 후보 랭킹** (예: "공기청정기 호환 필터를 하자").
- **사람** — 랭킹과 근거를 보고 **카테고리 1개 선택**.
- **Layer 2 (SKU 세부 검증)** — 선택된 카테고리 *안에서* 개별 SKU를 6단계로 검증. 출력 = **SKU 의사결정 도시에**.

### 1.2 핵심 설계 원칙 (실수 방지의 뿌리)
- **게이트와 점수를 분리한다.** 마진 30% 미만은 즉시 탈락(하드 게이트). 광고의존도 높은 카테고리도 강한 필터.
- **단일 점수로 줄 세우지 않는다.** 역할(5가지)은 태그이며, 겹치는 역할 수로 우선순위.
- **데이터 부족은 통과시키지 않는다.** `보류(검증 필요)`로 분리. 조용한 PASS 금지.
- **LLM은 원본 데이터에만 근거한다.** 환각 금지 프롬프트 강제.

## 2. 전체 구조

```
[Layer 1] 카테고리 발굴            "어떤 제품군에 진입할까?"
   ├ 발굴 신호 8개 스캔
   └ 출력: 카테고리 후보 랭킹 (큰 시장 × 가치 싸움 우선)
        ↓ 카테고리 후보
[사람]   카테고리 1개 선택          예: "로봇청소기 소모품"
        ↓ 선택된 1개
[Layer 2] SKU 세부 검증 (6단계)    "그 안에서 무엇을·어떻게 팔까?"
   1 수집 → 2 수요검증 → 3 경쟁스캔 → 4 마진·인증 게이트 → 5 불만+역할 → 6 도시에
        ↓
   출력: SKU 의사결정 도시에 + 우선순위
```

## 3. Layer 1 — 카테고리 발굴 엔진 (최우선 구현)

### 3.1 발굴 패턴
주 엔진: **"널리 보유된 내구재 + 정기 교체 소모품"**
- 와이퍼·에어컨필터 = **자동차**의 소모품
- 브러시·필터·먼지봉투·물걸레 = **로봇청소기**의 소모품
- 리필 헤드 = **전동칫솔**의 소모품

→ 시스템은 *"이 구조를 가진 다음 기기"*를 찾는다. 호환 소모품이 주 엔진이지만, 5가지 역할 렌즈로 **비(非)소모품 기회(심리형 등)**도 함께 포착한다(예: 차량 흠집/유리 복원 케어 확장).

### 3.2 발굴 신호 8개

| # | 신호 | 측정 방법 | 데이터 소스 |
|---|---|---|---|
| 1 | 베이스 기기 인기/보유 규모 | 기기 베스트셀러 순위 + 절대 검색량 | 마켓 베스트셀러, 검색광고 키워드도구 |
| 2 | 소모품 존재 | 정기 교체 부품(필터·브러시·헤드·봉투·패드·카트리지) 유무 | 마켓, 1688/알리바바 |
| 3 | 정품 가격 갭 | 정품 소모품가 vs 호환 가능가 (**갭 클수록 기회**) | 마켓, 1688 |
| 4 | 재구매 주기 | 교체 빈도 (**짧을수록 회전 엔진**) | 제품 정보, 리뷰 |
| 5 | 호환 경쟁 강도 | 호환 소모품 시장 포화 / 공백 | 마켓 |
| 6 | 우리 역량 적합 | 중국 OEM 생산 가능 + 마켓/브랜드 적합 (거의 항상 충족) | 1688/알리바바 |
| 7 | **시장 규모/수요** | **절대 월간검색수**(PC+모바일) + (마켓 연동 시 리뷰수×평균가 매출 추정) | **검색광고 키워드도구**, 마켓 |
| 8 | **진입 가능성/광고의존도** | **경쟁정도(compIdx) + 노출광고수(plAvgDepth)** + 가성비순 vs 브랜드순 | **검색광고 키워드도구**, 마켓 |

> 신호 7은 정밀 TAM이 아니라 **추정치**임을 명시(외부에서 정확한 매출 불가).
>
> **중요 — 데이터 소스 정정**: 네이버 **데이터랩 API는 상대값(기간 최대=100)만 제공**하므로 시장 규모(절대량) 산출에 쓸 수 없다. 절대 월간검색수는 **검색광고 키워드도구**(`monthlyPcQcCnt`+`monthlyMobileQcCnt`)에 있다. 데이터랩은 **추세·계절성 판정 전용**으로만 사용. CPC 입찰가는 키워드도구가 주지 않으므로(별도 API 필요) **경쟁정도(compIdx)·노출광고수(plAvgDepth)**로 광고의존도를 측정한다.

### 3.3 진입 가능성 판정 — 4분면 (신호 8의 강한 필터)
**원칙: 우리는 기능·가격(가성비) 싸움에서 이기고, 광고·브랜드 싸움에서는 진다.**

| | 가치 싸움 (우리가 이김) | 광고 싸움 (우리가 짐) |
|---|---|---|
| **시장 큼** | **최우선 진입** (예: 가성비 호환 필터류) | **함정 · 필터아웃** (예: 화장품) |
| **시장 작음** | 틈새 (구색·마진형) | 무시 |

- **광고의존도 높음 → "진입 어려움" 강한 감점/필터.** 시장이 아무리 커도(좌상단) 제외.
- 광고의존도 측정 핵심: **경쟁정도(높음)·노출광고수가 많고**, *최저가가 안 팔리고 비싼 브랜드가 팔리면* = 마케팅 시장(우리 불리).

### 3.4 Layer 1 출력 — 카테고리 후보 레코드
```python
category_candidate = {
    "category_name": str,           # 예: "공기청정기 호환 필터"
    "discovery_pattern": str,       # 호환소모품 | 심리형 | ...
    "signal_scores": dict,          # 신호 1~8 점수
    "market_size_est": str,         # 대 | 중 | 소 (추정)
    "winnability": str,             # 4분면 위치: 최우선 | 틈새 | 함정 | 무시
    "why_opportunity": str,         # "왜 기회인가" 한 줄
    "entry_difficulty": str,        # 예상 진입 난이도
}
```
- **정렬: 우상단(큰 시장 × 가치 싸움) 우선.** 광고 싸움 카테고리는 필터/하단.

## 4. 사람의 카테고리 선택
- Layer 1 랭킹 + 각 후보의 근거(4분면 위치, why_opportunity, 진입 난이도)를 보고 **1개 선택**.
- 선택된 카테고리가 Layer 2의 입력이 된다.

## 5. Layer 2 — SKU 세부 검증 (6단계 깔때기)

선택된 카테고리 *안에서* 개별 SKU를 검증한다.

### 5.1 파이프라인 & 순서 근거
```
1 수집 → 2 수요검증 → 3 경쟁 구조 스캔 → 4 마진·인증 게이트 → 5 불만+역할(LLM) → 6 도시에
```
- **게이트(4)를 LLM(5)보다 앞에 둔다** — 비싼 LLM을 탈락 후보에 돌리지 않기 위해.
- **단, 마진(4)은 "달성 가능 판매가"가 필요**하고 이는 경쟁 가격 분포(3)에서 나온다 → 3→4→5 순서가 의존성상 옳다.

### 5.2 SKU 후보 스키마
```python
sku_candidate = {
    "candidate_id": str, "product_name_raw": str, "normalized_name": str,
    "category": str,                # Layer 1에서 선택된 카테고리
    "source": str,                  # coupang | naver | datalab | alibaba | 1688
    "observed_price_krw": float,    # 시장 관측 판매가
    "sourcing_price_cny": float,    # 1688/알리바바 매입가 (없으면 None → 보류)
    "moq": int, "review_count": int, "rating": float,
    "search_trend_slope": float,
    "collected_at": str,            # now_kst() 패턴
}
```
- SQLite 누적 스냅샷(리뷰 수 시점별 저장 → 증가속도 계산) + Google Drive 백업, ads-analyzer 패턴 재사용.

### 5.3 단계별 명세

**[1] 수집** — 선택된 카테고리 내 SKU를 4개 소스에서 수집·정규화(중복 제거).

**[2] 수요 검증** — 검색 추세·리뷰 증가속도·계절성. 검색≈0 그리고 증가속도≈0이면 하드 제거. 그 외 `demand_score` + 플래그(declining/seasonal). 임계값은 config.

**[3] 경쟁 구조 스캔** — 판매자 수, 가격 분포(min/median/max), 상위 상품 리뷰·평점, 브랜드 집중도. **달성 가능 판매가** = 경쟁가 중앙값(기본) 또는 `pricing_percentile`(config).

**[4] 마진·인증 게이트** — 숫자 정밀성 최우선
> 마진 계산식은 새로 만들지 말고 `cost_engine`(기존 원가 로직)을 재사용. 아래는 참조 정의이며, VAT는 **공급가액(VAT 제외) 기준으로 일관 처리**(이중계산 방지).
```
판매가(VAT 포함) P = achievable_price
순매출(공급가액)  R = P / 1.1
변동원가 C = 매입원가(CNY×환율) + 국제운송 + 관세 + 통관 + 인증비분할 + 국내물류 + 포장 + 채널수수료(R×율)
마진율 = (R − C) / R × 100
```
게이트 판정:
```
sourcing_price_cny is None      → "보류(매입가 필요)"     # 통과 아님
cert_status == "불가/금지"        → "탈락(인증)"
마진율 < 30 (margin_gate_pct)    → "탈락(마진)"            # 하드 게이트
else                            → "통과" → 5단계
```
> ※ 관세·고정단가(③⑥)가 아직 미입력이므로, 마진 게이트는 **간이 추정 모드**로 동작한다(8절 참조). 정밀 마진은 실제 진입 직전 ③⑥ 입력 후.

**인증 분류기 (자문용, 7절 ⑤ 룰 사용):** 카테고리 → 인증 요건 매핑. 출력에 항상 `검증 필요` 플래그. 시스템은 법적 확정 결론을 내리지 않는다.

**[5] 불만 마이닝 + 역할 분류 (LLM, 생존 SKU만)**
- *불만 마이닝*: 상위 경쟁 상품 리뷰에서 반복 불만/미충족 니즈 추출 → `opportunity_hypothesis`(개선 스펙 가설). 프롬프트 제약: 실제 리뷰에 있는 것만, 부족하면 "근거 부족", 추측 금지.
- *역할 분류 (5가지)*:

| 역할 | 판별 | 신호 |
|---|---|---|
| 회전형 | 수치 | 리뷰 증가 빠름 + 재구매 주기 짧음 + 검색 꾸준 |
| 마진형 | 수치 | 공헌이익률 ≥ **50%** (`margin_high_pct`) + 가격경쟁 약함 |
| 구색형 | 수치(근사) | 기존 주력 SKU와 카테고리/용도 인접성 + 진입장벽 낮음 |
| 콘텐츠형 | LLM | before/after 시연 가능, 시각 변화 큼, 후킹 명확 |
| 심리형 | LLM | '비싼 대안'을 싼 값에 대체(복원 vs 교체), 불안·비용회피·욕구 |

- `role_overlap_count` = 해당 역할 수. **겹칠수록 고확신**(예: 유리복원제 = 심리+콘텐츠).

**[6] 의사결정 도시에 + 우선순위**
- 도시에 항목: 상품명/출처, **역할 태그+겹침 수**, 수요 요약, 경쟁 구조, **마진(원가 분해)**, 인증(`검증 필요`), **불만/기회 가설**, 소싱 후보(공급사·가격·MOQ), 리스크 플래그(인증·계절성·데이터공백·IP·**표시광고법**).
- 정렬: ①`role_overlap_count` 내림차순 → ②마진율 내림차순. **단일 은닉 점수로 회전형/마진형을 섞어 비교 금지.** 역할별 필터·정렬 제공.
- 출력: Streamlit 대시보드 + xlsx/CSV. 표시 숫자 전부 반올림.

## 6. 정밀성·실수 방지 규칙

1. **숫자 정합성**: 금액 계산은 `cost_engine` 재사용. VAT 공급가액 기준. 환율·관세·수수료·물류비는 config.
2. **데이터 부족 처리**: 매입가/리뷰 없음 → `보류`. 조용한 PASS 금지.
3. **법적 판단 회피**: 인증·표시광고법 출력은 자문 플래그.
4. **환각 방지**: LLM 단계는 원본 데이터 범위 내. 불만 마이닝은 실제 리뷰 근거.
5. **게이트 vs 점수 분리**: 30% 마진(SKU) + 광고의존도(카테고리)는 게이트/필터. 역할은 태그.
6. **표시 반올림**: 화면 숫자 전부 round/toFixed/천단위.
7. **간이 추정 모드**: ③⑥ 미입력 시 마진 게이트는 보수적 추정으로 동작하며, 도시에에 "마진=추정치" 플래그 부착.

## 7. 반영된 설정 (Resolved)

- **① 데이터 수집 방식**: **공식 API 우선** → 미제공/불가 소스는 자동 스크래핑 또는 수동, **최대한 자동화**. 데이터랩·네이버 검색광고 = API 확정. (소스별 구체 방식은 구현 시 확정 — 12절 #1)
  - **현재 구현 상태(네이버)**: 라이브 HMAC API 연동 전, **CSV 브리지**로 먼저 검증 중. 오빠두 '네이버 검색광고 연관검색어 스크랩' 엑셀에서 내보낸 CSV(`검색키워드`/`연관키워드`/`월검색수(PC·모바일)`/`월평균노출광고수`/`경쟁정도`)를 `CSVAdapter`가 읽어 신호 7·8로 매핑. 동일 API·동일 필드이므로, 검증 후 라이브 `NaverAdapter`(HMAC 자동 호출)로 어댑터만 교체.
- **② 채널 수수료**: **플랫폼 + 카테고리를 선택하면 유지 테이블에서 요율이 자동 입력**되는 UI 방식. (요율 값 자체는 대표님 입력 — 12절 #2)
- **④ 마진형 기준** `margin_high_pct` = **50%**.
- **⑤ 인증 분류 룰** (근거: 『생활화학제품 및 살생물제의 안전관리에 관한 법률』, 현행 고시 기후에너지환경부고시 제2025-31호, 시행 2025.11.24 / 14분류 43품목):

| 제품군 | 인증 분류 | 비고 |
|---|---|---|
| 워셔액 / 부동액 | **안전확인대상**(자동차용제품) — 신고 + 특수검사 | 명시 품목 |
| 광택제 / 발수코팅제 | **안전확인대상 가능성 높음**(코팅제품) → 확인 | 광택코팅제·특수목적코팅제 |
| 유리복원제 / 유리세정제 | 세정 vs 코팅·연마 분류 의존 → **확인** | 성분·기능별 |
| 엔진오일 | **생활화학제품 아님**(석유사업법 등 타 법령) | 별도 규제 확인 |
| 와이퍼 / 에어컨필터 / PPF·보호필름 | 화학 인증 불요(물리 제품) | PPF 시공젤 등 화학 부자재는 별도 |
| 전기 / 배터리 제품 | **KC 인증**(전기용품 및 생활용품 안전관리법) | 기존 배터리 KC 프로그램 참고 |

> 룰 원칙: ⓐ 타 법령으로 규제되면 생활화학제품 대상 제외(엔진오일). ⓑ 소비자 판매 제품은 대상(정비소 작업공간 전용만 제외). 모든 "대상/확인"에는 `검증 필요` 플래그. 최종 확인은 KTR 등 시험·검사기관 → 한국환경산업기술원 신고(확인일로부터 30일 이내, 3년마다 갱신).

## 11. 검증 / 테스트 체크리스트 (`pytest`, ads-analyzer 컨벤션)

**Layer 1**
- 발굴 패턴이 알려진 사례(로봇청소기 소모품, 전동칫솔 헤드)를 재현하는가.
- 광고 싸움 카테고리(화장품류)가 신호 8로 필터/하단 처리되는가.
- 4분면 분류(최우선/틈새/함정/무시) 정확.

**Layer 2 — 마진(치명 구간)**
- 알려진 상품: 입력 → 기대 마진율 일치.
- 경계값: 마진 정확히 30.0% → 통과/탈락 경계.
- 결손값: `sourcing_price_cny=None` → `보류` 반환(통과 아님).
- 0 division: `예상판매수량=0` → 예외 처리.
- VAT 공급가액 기준 일관성(이중계산 없음).
- 간이 추정 모드: ③⑥ 미입력 시 "추정치" 플래그가 도시에에 부착되는지.

**파이프라인**
- 게이트가 LLM보다 먼저 실행(탈락 후보 LLM 미호출).
- 역할 겹침 수 내림차순 정렬.
- LLM 불만 마이닝: 리뷰 부족 시 '근거 부족' 반환.
